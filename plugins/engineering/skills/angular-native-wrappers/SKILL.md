# Wrapping Imperative APIs in Angular

You have a large, imperative, stateful library — a map, an editor, a chart, a media player — with dozens of methods and events. The naïve approach hand-writes a wrapper method per native method, a manual output per event, and manual lifecycle/zone glue. That is pure repetition (`architect-mentality` → *Work smart, not hard*; *Automate every repeated process*), it breaks every time the SDK changes, and it bloats the code.

This skill is the pattern that makes the wrapping **generated by convention**, **fully typed**, and **change-detection-correct** — the complexity paid for once, centrally (*Concentrate complexity*), so each concrete component is just declarations.

Adopt the pieces **together**; they reinforce one another.

## The model

```
native object        →  the raw SDK instance (google.maps.Map, an editor, a chart…)
  ⤷ wrapper          →  typed; delegates to native by convention; owns lifecycle + CD policy; exposes .native
      ⤷ component    →  the declarative Angular surface: inputs, outputs
          ⤷ plugins  →  optional, tree-shakable capabilities (see angular-architecture → extensibility)
```

---

## 1. Delegate to the native object by convention — don't hand-write pass-throughs

**What.** Instead of `setZoom(z) { this.native.setZoom(z); }` a hundred times, intercept member access on the wrapper and delegate to the native object automatically.

**Why.** *Work smart* + *Automate every repeated process* + *Concentrate complexity*: delegation exists once; a new native method needs **zero** wrapper code; an SDK upgrade costs ~nothing.

```ts
export abstract class NativeWrapper<TNative extends object> {
  constructor(public readonly native: TNative) {
    return new Proxy(this, {
      get: (target, prop, receiver) => {
        if (prop in target) return Reflect.get(target, prop, receiver);   // hand-written members win
        const member = (this.native as any)[prop];
        return typeof member === 'function' ? member.bind(this.native) : member;
      },
    });
  }
}
```

**When NOT to use.** A small SDK with a handful of methods — just write thin wrappers; this machinery is overkill (*Know when not to do it*). The pattern earns its keep only on large imperative surfaces.

## 2. Keep full type safety — derive the surface, merge it in

**What.** The Proxy supplies methods at runtime with no compile-time signatures. Restore IntelliSense by deriving an interface from the native type with mapped/conditional types and **merging it into the class via a same-name interface** (declaration merging).

```ts
type NativeMethods<T> = { [K in keyof T as T[K] extends Function ? K : never]: T[K] };

export interface GoogleMap extends NativeMethods<google.maps.Map> {}   // phantom typed surface
export class     GoogleMap extends NativeWrapper<google.maps.Map> {}    // methods exist at runtime via the Proxy
```

**Why.** *Refuse false tradeoffs* — dynamic delegation **and** full types, not either/or.

**Keep runtime and type exclusions in lockstep:** if you forbid `setMap` at runtime, also `Omit` it from the derived interface, so the compiler blocks what the runtime would reject. (This is deep TypeScript; it will move into the future `advanced-typescript` skill and be cross-linked from here.)

## 3. Encode policy as metadata, not scattered conditionals

**What.** Which methods must run *without* triggering change detection, which are excluded — declare per method as data, read by the delegation layer.

```ts
@NativeWrapperConfig<GoogleMap>({ panBy: 'outside-cd', setMap: 'exclude' })
export class GoogleMap extends NativeWrapper<google.maps.Map> { /* … */ }
```

**Why.** *Place everything on purpose* + *Concentrate complexity*: policy is data in one place, not `if`s sprinkled through the delegation path. Convention for the common cases (e.g. `get*` delegates as-is, `set*` runs without CD); the decorator overrides the exceptions.

## 4. Auto-wire the component surface: inputs → native, native events → outputs

**What.** A component base maps each input to the native setter by name, and each native event to an output — so a concrete component just **declares** properties and events.

Modern Angular (signals): reflect input signals onto the native object with `effect()`; surface native events through `fromEventPattern` so RxJS owns listener teardown.

```ts
@Directive()
export abstract class NativeComponentBase<TWrapper extends NativeWrapper<any>> {
  readonly wrapper = inject<TWrapper>(WRAPPER);
  private readonly destroyRef = inject(DestroyRef);

  /** input signal → native setter, reactively. Call from the constructor (injection context). */
  protected bindInput<V>(value: Signal<V>, nativeSetter: keyof TWrapper): void {
    effect(() => (this.wrapper as any)[nativeSetter]?.(value()));
  }

  /** native event → auto-unsubscribed observable */
  protected event<T>(name: string): Observable<T> {
    return fromEventPattern<T>(
      handler => this.wrapper.native.addListener(name, handler),   // adapt to the SDK's listener API
      (_, listener) => listener.remove(),
    ).pipe(takeUntilDestroyed(this.destroyRef));
  }
}
```

(Old Angular used `ngOnChanges` + `EventEmitter`; the timeless idea — *declarative surface, framework owns the wiring* — is unchanged.)

**Why.** *Work smart* + *Design for the consumer*: a new component is just declarations; no per-property glue.

---

## 5. Control when Angular reacts (zone / change-detection discipline)

**What.** Native mutations and high-frequency callbacks (drag, mousemove, render ticks) must **not** thrash change detection; user-facing events that should update the view **must**.

- **Zone-based apps:** run mutations/creation in `NgZone.runOutsideAngular`; re-enter via `ngZone.run(...)` for events that update the view.
- **Zoneless / OnPush** (the modern direction): nothing auto-runs CD — reflect native state into the view through **signals**, and update only the signals you actually render.

**Why.** *Compensate for your materials' weaknesses* — high-frequency imperative callbacks are a known CD footgun; decide the boundary by *category* (mutations/creation = no CD; user-facing events = update the view) and enforce it at one choke point, not per call site.

---

## 6. Async readiness — load once, gate rendering, then write synchronous code

**What.** Many SDKs load via an async script. Load it lazily, publish readiness as a promise/signal through a DI token, and **gate** the subtree until ready — so all downstream code may assume the native API exists.

```ts
@Directive({ selector: '[whenMapReady]' })
export class WhenMapReadyDirective {
  constructor() {
    const tpl = inject(TemplateRef);
    const vcr = inject(ViewContainerRef);
    inject(MapLoader).ready.then(() => vcr.createEmbeddedView(tpl));
  }
}
```

**Why.** *Compensate for weaknesses* + *Concentrate complexity*: solve async-readiness **once** at a boundary instead of threading promises/observables through the whole codebase.

---

## 7. Never trap — always expose `.native`

The wrapper must expose the raw native object (typed) as a **documented** escape hatch, stating that mutating it directly bypasses the wrapper's tracking and CD handling. (`architect-mentality` → *Abstractions must never trap*.)

---

## When to reach for this skill

- ✅ A large imperative / stateful / third-party JS API you must present as Angular components.
- ❌ A tiny API (few methods, few events) → write thin wrappers; this machinery is overkill (*Know when not to do it*).

## Pitfalls

- Returning a `Proxy` from the constructor changes instance identity — keep the wrapper's own methods on the prototype and let the trap prefer them (piece 1).
- Drift between type-level exclusions and runtime exclusions → compiler and runtime disagree (piece 2).
- In zoneless apps, don't assume CD runs after a native callback — drive the view via signals (piece 5).
- Don't let the convenience surface hide the `.native` escape hatch (piece 7).
