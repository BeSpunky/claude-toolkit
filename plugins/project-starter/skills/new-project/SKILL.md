# Start a new project (BeSpunky standard)

Scaffold a new project into the projects root (see step 1).

## Prerequisite (do not skip)

The `project-starter@claude-toolkit` plugin must be installed and **current** in the session running this skill — otherwise this SKILL.md isn't even loaded and the procedural details below aren't in your context. If you have this skill available (you're reading this), good. If you've been asked to scaffold but this skill is **not** available in your session, ask the user to install/refresh the plugin first (`/plugin marketplace update claude-toolkit` then `/reload-plugins`), then re-trigger the scaffolding request. **Do not improvise scaffolding from memory** — every step below encodes a hard requirement and dropping any one of them produces a non-compliant project.

## Hard requirements (do not deviate)

- **Nx-first**, created via `yarn create nx-workspace` (latest Nx, no version pinning).
- **Integrated monorepo** (`apps/` + `libs/`), via `--preset=apps` then `nx add @nx/angular`. (A literal `angular-*` preset always forces a demo app and is verified impossible to suppress; `apps` + `nx add @nx/angular` is the only one-shot Angular-ready path with no demo.)
- **Angular** wired in, with **one clean `--minimal` app** (no Nx welcome/demo content).
- **Dev server reachable from outside the devcontainer**: the app's `serve` target gets `host: 0.0.0.0`. Reliable file-watching over WSL/Docker mounts is handled by `CHOKIDAR_USEPOLLING` + `CHOKIDAR_INTERVAL` env vars set on the **devcontainer** (covers any chokidar-based watcher across old and new Angular builders). Modern Angular's `@angular/build:dev-server` does **not** accept a `poll` option, which is why polling lives in the devcontainer env, not on the serve target.
- **Node from the devcontainer base image, via Docker** - never nvm. The script resolves the newest `mcr.microsoft.com/devcontainers/typescript-node:<major>` tag and uses it both to run the scaffold and as the devcontainer image (single source of truth for Node).
- **Devcontainer** with the **Claude CLI** (feature) and the **Claude VS Code extension** (`Anthropic.claude-code`), plus the `.claude` persistence mount.
- **CLAUDE.md** capturing the project's intentions, with the `engineering` plugin's always-on directives (**Architect mentality** + **Architecture-first**) verbatim and a **toolkit version stamp** at the top.
- **bespunky/claude-toolkit** marketplace wired into the new project's `.claude/settings.json` with **`autoUpdate: true`** (toolkit refreshes at startup), AND pre-installed by the devcontainer's `postCreateCommand` (`claude plugin marketplace add` + `claude plugin install --scope project`) for `project-starter@claude-toolkit`, `engineering@claude-toolkit`, **and `browser-automation@claude-toolkit`**, so the toolkit's skills/agents are live the moment the container builds - no manual install. (Declaring `enabledPlugins` alone does not auto-install; the CLI step is what makes it immediate. Note: for third-party marketplaces the guaranteed auto-update switch is the `/plugin` UI toggle / managed settings.)
- **Angular AI tooling (always on)** — the scaffolder runs the house `angular-ai` generator, which wires Angular's official AI tooling in two layers. (1) **MCP**: a project-scoped, committed `.mcp.json` declaring the **Angular CLI MCP server** (`npx -y @angular/cli mcp` — always the latest CLI). Only the **default knowledge tools** are enabled (`get_best_practices`, `search_documentation`, `find_examples`, `ai_tutor`, `onpush_zoneless_migration`); the **experimental exec tools** (`build`, `devserver.*`, `test`, `e2e`) are deliberately omitted because an integrated Nx workspace has **no root `angular.json`** — they read `angular.json` / shell out to `ng build` and would only pull agents off Nx-owned targets (Nx + the Playwright skill own build/serve/test verification). The generator **merges** into any existing `.mcp.json`, preserving user-added servers. (2) **Skills**: the Angular **agent skills** (`angular-developer`, `angular-new-app`) are **not vendored** — `post-create.sh` fetches them fresh from `github.com/angular/skills` into `.claude/skills/` on every container build (gitignored cache; the skills repo is itself auto-generated upstream, so a fork would silently go stale). **Not opt-in** — every BeSpunky project gets it (the skills are Claude's own `SKILL.md` format; MCP is the live knowledge channel, the skills the always-loaded companion).
- **Playwright (always on)** — the scaffolder runs the house `playwright` generator which adds `@playwright/test` to `devDependencies`, and the devcontainer's `post-create.sh` detects that entry and runs `playwright install --with-deps chromium` (Chromium binary + apt deps) on container build. The browser binary is cached in a per-workspace Docker volume so rebuilds reuse it. Pairs with the `browser-automation:playwright` skill (loaded via the plugin above) which describes when to use Playwright vs. the preview / Chrome MCPs, how to write headless scripts, and how to feed screenshots back. **Not opt-in** — every BeSpunky project gets it (bounded cost: ~150 MB cached browser, ~2 min one-time install; the payoff is "Claude can verify UI changes end-to-end the moment the container opens").
- **Private GitHub repo (always-on)** — full scaffold creates a **private** GitHub repo via `gh` and pushes the project to it. This runs **host-side, after** the Docker scaffold (the `gh` auth lives on the host, not in the bare base image), so the host running the script must have `gh` installed and authenticated (`gh auth login`). It is **best-effort, never fatal**: if `gh` is missing/unauthenticated, or an `origin` remote already exists, the script logs a `GITHUB_SKIP` reason and the (fully committed) local repo is left intact. Opt out with **`--no-github`**. Repair mode never touches the remote. **Why always-on, not just for Firebase:** Firebase App Hosting deploys are GitHub-driven — linking the repo at `firebase apphosting:backends:create` is what makes Firebase provision its own Cloud Build CI/CD, so the repo must already exist; and non-Firebase projects still want a remote. The scaffolder generates **no** deploy workflow of its own — see the Firebase bullet.
- **Firebase support is opt-in**: when scaffolded with `--firebase`, the scaffolder wires Firebase end-to-end:
  - **Devcontainer**: adds the **Firebase CLI** (`ghcr.io/devcontainers-extra/features/firebase-cli`) + **Google Cloud CLI** (`ghcr.io/jajera/features/gcloud-cli`) features, the `toba.vsfire` VS Code extension, labeled `portsAttributes` entries, and **explicit same-port `forwardPorts`** for the dev server + emulator suite (4200, 4000 UI, 9099 Auth, 8080 Firestore, 9150 ws, 9199 Storage, 5001 Functions). Same-port forwarding is **required** for the app to work in a host browser: the page loads over forwarded :4200, then the Firebase SDK *inside that page* calls the emulators at the hardcoded `localhost:9099`/`localhost:8080` from `environment.ts` — addresses that only resolve from the host when those container ports forward to **identical** host ports (auto-forward may pick a different free port, which the browser SDK can't discover → `ERR_CONNECTION_REFUSED` on auth refresh → Firestore writes hang). Tradeoff: parallel Firebase devcontainers collide on these host ports; to run several, re-port one container in `devcontainer.json` + `environment.ts` + `firebase.json` together. Backend emulators are `onAutoForward: "silent"`, the dev server `openPreview`.
  - **Emulator config**: writes **`firebase.json`** — the emulator suite (+ `singleProjectMode: true`) **and a `functions` block** pointing at the built Nx output (`dist/apps/functions`, predeploy = `yarn nx lint functions && yarn nx build functions`). The functions block is **required**: a configured functions emulator with no backend behind it makes `emulators:start` fatally abort. The `emulators`/`functions` keys are generator-owned; other top-level keys the user adds (rules paths, …) are preserved. The generator does **not** write `.firebaserc` — the cloud-project linkage belongs to the Firebase CLI (`firebase use --add` validates against the user's actual account). The launch script passes `--project=demo-<workspaceName>` explicitly so emulators work without `.firebaserc`, without login, and without a real GCP project (the `demo-` prefix is Firebase's documented offline-only convention). Also adds the emulator `.gitignore` entries (`*-debug.log`, `firebase-debug.*.log`, `/.emulator-data`) and disables the Nx TUI (`nx.json` → `tui.enabled: false`) so the continuous emulators + dev-server pair streams plain prefixed logs.
  - **Cloud Functions as a first-class Nx app** (`apps/functions`): esbuild-bundled by `nx build functions` into `dist/apps/functions` with a generated deploy-manifest `package.json`; runtime deps (`firebase-admin`/`firebase-functions`) live at the **workspace root** (no per-project `node_modules`); lints via the workspace flat config; `nx run functions:deploy` runs `firebase deploy --only functions`. Source files are written only if absent (the user owns their functions); the build/lint/deploy targets are generator-owned. A **platform firewall** is inserted into the root ESLint config's depConstraints: `platform:web` (the app) bans `firebase-admin`/`firebase-functions` imports; `platform:server` (`functions`, `firebase`) bans `firebase`/`@angular/*`.
  - **The emulator suite as its own workspace-level Nx project** (`firebase/project.json`): targets `emulators` (full suite, `dependsOn` functions:build), `emulators:auth`/`:firestore`/`:storage`/`:functions` (one + UI), `seed:build`, and `reset` — all funneling through **`tools/emulators.sh`** (the single launch path: **reap → prime → start**). Reap = `tools/reap-emulators.sh`, a verified process+port reclaim (kills orphaned emulator JVMs by cache path, then polls the configured ports until actually free, SIGTERM → SIGKILL) so an ungraceful prior death never breaks the next start. Prime/start = the **seed & cache layer**: the suite imports the gitignored `.emulator-data/` working dir (primed from a committed seed in `tools/emulator-seeds/` on first run) and exports back on clean exit (full runs only) — onboard once, session + data survive every serve. Seeds are generated from the **declarative, user-owned `tools/seed/world.mjs`** via `nx run firebase:seed:build`; reset on call with `nx run firebase:reset`.
  - **App initialization** (Angular's environment-files pattern, **per-service emulator-aware**): generates **`environment.interface.ts`** (shared shape; rewritten every run), **`environment.ts`** (dev — `production: false`, `authDomain: 'demo-<workspaceName>.firebaseapp.com'`, and a **per-service emulator toggle**: an `EMULATE` map drives each service's emulator `default`), **`environment.no-emulators.ts`** (dev with **no emulators block** — compiled by `serve-no-emulators`, so that target connects nothing to emulators), and **`environment.prod.ts`** (production). The dev env files are written **only if absent** (the legacy single-string-emulators `environment.ts` is upgraded to the per-service shape). The production build swaps in `environment.prod.ts`, and a **`no-emulators` build configuration** swaps in `environment.no-emulators.ts` (both registered idempotently). **`emulator-overrides.ts`** (generator-owned) is the per-session override resolver — `?emulate=`/`?real=`/localStorage, precedence *default < localStorage < URL* — so a dev can run e.g. **Firestore emulated + real Auth** without editing source or rebuilding. **`firebase.config.ts`** exports `provideAppFirebase()`, gating **each service independently** on `committed-default ⊕ override`, all behind **`ngDevMode`** — Angular's dev-mode flag, which the optimizer folds to `false` in production builds, so the resolver, every `connect*Emulator(...)` call, the endpoints, and the whole override module tree-shake out of the prod artifact. (We gate on `ngDevMode`, **not** `environment.production` — the latter, a const-object property, isn't reliably inlined by the esbuild builder and leaves emulator code in prod; verified by grepping the production bundle. Firebase's own SDK still ships its internal emulator support — that's library code, not ours.) The fail-loud prod guard lives here too (and stays on `environment.production`, since it SHOULD ship in prod). **`firebase.config.ts` is generator-owned by default** — written when absent and rewritten on outdated shapes (the ancient two-consts shape, the pre-per-service whole-block shape, OR a per-service file that predates the `ngDevMode` prod-DCE gate; `--repair --firebase` self-heals all of them, with a logged warning to re-apply any custom provider wiring). Once it's on the current shape (imports `'./emulator-overrides'` **and** uses the `ngDevMode` gate) it's left alone.
  - **App wiring**: best-effort insertion of `provideAppFirebase()` into `apps/<app>/src/app/app.config.ts`'s providers array (logs a warning if the file shape is unrecognized — you then add it manually).
  - **Nx targets** on the app — **three single-purpose serve targets, so local dev is never forced through the emulators and no target carries a confusing "other-mode" config**: **`serve`** is an `nx:run-commands` **orchestrator** running `firebase:emulators` + **`serve-with-emulators`** in parallel — one command, one Ctrl+C, the full local Firebase stack (the default `nx serve <app>`); **`serve-with-emulators`** is the app dev-server pinned to the emulator env (`build:development`); **`serve-no-emulators`** is the app dev-server pinned to the no-emulator env (`build:no-emulators`) — `nx run <app>:serve-no-emulators`, the app against a real/staging backend (or pure UI work). Both dev-servers pin their env via `options.buildTarget` and carry **no `configurations`**, so a `no-emulators` target can never be asked to serve *with* emulators (and vice-versa). The per-app `emulators*` targets of earlier generations moved to the workspace-level `firebase` project (run/restart the suite alone via `nx run firebase:emulators`). **Repair self-heals every earlier generation**: the app dev-server — a fresh `serve`, or a legacy inner target (`serve-app`/`serve-standalone`, or the old `serve-no-emulators` that doubled as the orchestrator's inner dev-server with development/production/no-emulators configs) — is reshaped into this trio; the legacy/duplicate names and their stale configs, plus stray app-level `emulators*` targets, are removed.
  - **Deps**: the generator adds `firebase` + `@angular/fire` (dependencies), `firebase-admin` + `firebase-functions` (the Functions runtime, at the **workspace root** — keep aligned with `apps/functions/package.json`, the deploy manifest), and `@nx/esbuild` (devDependency, pinned to the workspace's own Nx version — Nx plugin packages move in lockstep with `nx`) to `package.json`, then runs the package-manager install via Nx's `installPackagesTask` (post-commit). Existing entries are never overwritten (user pins survive `--repair`). No shell-side `yarn add`.
  - **CI/deploy is Firebase's, not ours**: the scaffolder writes **no GitHub Actions deploy workflow**. Deploy CI is wired when the user links the (always-created, see the GitHub bullet above) repo at `firebase apphosting:backends:create` — App Hosting then provisions and maintains its own Cloud Build pipeline and auto-deploys on every push to the configured branch. This is deliberate: relying on Firebase's native GitHub integration means deploys keep working as Firebase evolves its mechanism, with nothing in the project to maintain. The repo-as-CI linkage is documented in the "connect a real Firebase project" recipe below, the `{{#firebase}}` block of `CLAUDE.md.tmpl`, the `apphosting.yaml` header comment, and the devcontainer welcome banner.
  - **Never enabled by default** — only when the user explicitly asks for Firebase or **you asked and they said yes**.

## Generator-first, manual last (applies to ALL scaffolding)

For anything Nx can generate - apps, libraries, components, services, project config - **use the Nx generator (`nx g ...`); never hand-create and fill files.** Before hand-writing anything structural, check what exists: `nx list <plugin>` / `nx g <generator> --help` (or the `nx-generate` skill / the Nx MCP server). Only fall back to manual file creation when no generator covers the task. **Never guess flags** - verify against `--help` / `nx_docs`.

In this skill that means: the workspace, the Angular plugin, the app, the serve options, the devcontainer, the Claude settings, and the Playwright devDep all come from generators - the `@nx/*` ones plus the house `@bespunky/nx-tools` generators (`app` — which wraps `@nx/angular:application` and applies the per-app config, so it's the single command for the first app *and* every later one —, `serve-options`, `devcontainer`, `claude-settings`, `angular-ai`, `playwright`). The **only** non-generated artifact is the tailored `CLAUDE.md`, which the skill authors from the project's intentions.

## 1. Run the scaffold script

The bundled script is at **`${CLAUDE_SKILL_DIR}/assets/scaffold.sh`** (that variable resolves to wherever this skill is installed - never hardcode a path). It performs three phases inside the devcontainer base image (via Docker, no nvm). **Run the whole script** - do not replicate its steps by hand; the phases together are the project's hard requirements:

- **1.1 Bootstrap Nx + Angular + the app**: `yarn create nx-workspace … --preset=apps` → `nx add @nx/angular` → `nx g @bespunky/nx-tools:app apps/<APP> [--firebase=true]`. The app is created by the **house `app` generator**, not raw `@nx/angular:application`: it delegates to `@nx/angular:application` with the house defaults (`--minimal --style=scss --routing --e2eTestRunner=none`) **and** applies the per-app config (serve `host: 0.0.0.0` via `serve-options`, plus the full Firebase wiring via `firebase-emulators` when `--firebase=true`). This is the **same one command a developer runs to add any later app** (`nx g @bespunky/nx-tools:app apps/<name>`), so the first app and every subsequent app are configured by one code path and can't drift. Firebase is passed explicitly here because `firebase.json` doesn't exist yet at first-app time; for a *later* app the generator auto-detects it.
- **1.2 Install the house plugin (`@bespunky/nx-tools`)** into the workspace's `node_modules` and compile its TypeScript generators to JS (so Nx can load them from `node_modules`).
- **1.3 Run the workspace-level house generators** (Nx-native via devkit `Tree`, format-aware, **all idempotent**). The **per-app** generators (`serve-options`, and `firebase-emulators` under `--firebase`) are not invoked here — the **1.1 `app` generator already applied them to the app**; the generators below are workspace-wide and run once (`--repair` re-runs the per-app ones directly against the existing app, see §1a):
  - `nx g @bespunky/nx-tools:playwright` → adds `@playwright/test` to `devDependencies` (the devcontainer's `post-create.sh` installs Chromium + apt deps on build).
  - `nx g @bespunky/nx-tools:devcontainer --name=<PROJECT> --nodeMajor=<MAJOR>` `[--firebase=true]` → writes `.devcontainer/devcontainer.json` (Claude CLI feature, VS Code extension, `.claude` persistence mount, polling env vars, `postCreateCommand` pre-installing the toolkit plugins; with `--firebase=true` adds the firebase-cli + gcloud-cli features, `toba.vsfire`, labeled `portsAttributes`, and **explicit same-port `forwardPorts`** for the dev server + emulator suite — required so the Firebase SDK in a host browser can reach the emulators at their hardcoded `localhost:<port>` addresses).
  - `nx g @bespunky/nx-tools:claude-settings` → writes `.claude/settings.json` (marketplaces + `autoUpdate: true` + `enabledPlugins`) and keeps `.claude/data` out of git.
  - `nx g @bespunky/nx-tools:angular-ai` → writes/merges project-scoped `.mcp.json` declaring the Angular CLI MCP server (`npx -y @angular/cli mcp`, default knowledge tools only — no experimental exec tools, which don't fit an Nx workspace) and gitignores `.claude/skills/` (the build-time-fetched Angular agent skills cache). The skills themselves are fetched by `post-create.sh`, not this generator.
  - **Only with `--firebase` (applied per-app by the 1.1 `app` generator — which resolves `--workspaceName` from the workspace root, never the app name; `--repair --firebase` runs it directly against an existing app):** the `firebase-emulators` generator writes `firebase.json` (emulators + functions blocks; the generator does **not** write `.firebaserc` — that's the Firebase CLI's job; see the "Connect to a real Firebase project" recipe below), the emulator `.gitignore` entries, `nx.json` TUI-off, the environment files + `apps/<APP>/src/app/firebase.config.ts` (the `provideAppFirebase()` factory), wires `app.config.ts`, scaffolds **`apps/functions`** (Cloud Functions Nx app) and **`firebase/project.json`** (the workspace-level emulator project: `emulators`, `emulators:<svc>`, `seed:build`, `reset`), writes the emulator tooling (`tools/emulators.sh`, `tools/emulator-data.sh`, `tools/reap-emulators.sh`, `tools/seed/*`), reshapes the app's serve into **three single-purpose targets** — `serve` = the parallel orchestrator (`firebase:emulators` + `serve-with-emulators`), `serve-with-emulators` = the app dev-server pinned to the emulator env, and `serve-no-emulators` = the app dev-server pinned to the no-emulator env (`nx run <app>:serve-no-emulators`), each with no sub-configs — inserts the platform-firewall ESLint constraints, AND adds `firebase` + `@angular/fire` + `firebase-admin` + `firebase-functions` + `@nx/esbuild` to `package.json` (the generator's post-commit `installPackagesTask` runs the install). No shell-side `yarn add`.
- **1.4 Commit + create the private GitHub repo** (scaffold mode only, unless `--no-github`): the script commits the full scaffold (the generator outputs land *after* `yarn create nx-workspace`'s initial commit), then — **host-side, after the Docker phases** — runs `gh repo create <PROJECT> --private --source <path> --remote=origin --push`. The `gh` step runs outside the container because the host holds the `gh` auth; it's best-effort and never fails the scaffold (logs `GITHUB_SKIP` with a reason if `gh` is unauthenticated or `origin` already exists).

Projects root:
- Default: **`$HOME/projects`** (resolves to the current user's home at runtime).
- Override: export **`PROJECTS_DIR=/some/other/root`** before running.

Invoke it according to the environment you're running in:

- **Windows host where Docker runs inside WSL** (the common BeSpunky setup): `${CLAUDE_SKILL_DIR}` is a Windows path, and Docker/the projects root live in WSL - so run the script inside WSL, converting the path with `wslpath`:

- **Linux/macOS, or inside a devcontainer** where Docker is local:
  ```
  bash "${CLAUDE_SKILL_DIR}/assets/scaffold.sh" [--firebase] [--no-github] <PROJECT_NAME> [<APP_NAME>]
  ```

Include `--firebase` **only** if step 0 input 4 was yes (the user opted in). Omit it otherwise. Include `--no-github` **only** if the user explicitly opted out of a remote (otherwise a private GitHub repo is created).

Wait for the final `SCAFFOLD_OK <path>` line. It also reports the GitHub outcome — `GITHUB_OK <url>` when the private repo was created and pushed, or `GITHUB_SKIP: <reason>` when it was skipped (so you can relay the reason and the manual follow-up). First run pulls the base image (a few hundred MB). If a generator/CLI flag is rejected, check `yarn create nx-workspace --help`, `nx g @nx/angular:application --help`, or `gh repo create --help` - **never guess**.

### 1a. Recovery (if the script can't complete end-to-end)

If `scaffold.sh` fails partway, **do not improvise the remaining phases by hand** - the house generators encode hard requirements that are easy to miss. Use the script's **repair mode**, which re-runs **only** phases 1.2 and 1.3 on the existing project (all generators are idempotent):

```
bash "${CLAUDE_SKILL_DIR}/assets/scaffold.sh" --repair [--firebase] <PROJECT_PATH_OR_NAME> [<APP_NAME>]
```

(Through WSL on Windows hosts, wrap as in step 1.) `--firebase` and `--repair` may be combined to retrofit Firebase support onto an existing project. If repair errors with "node_modules/.bin/nx not found," run `yarn install` in the project first, then re-run.

**Repair auto-backs up to git first.** Because repair re-runs the generators — which *rewrite* files (notably `firebase.config.ts` when it self-heals a legacy/pre-per-service shape) — `--repair` snapshots the project **before** touching anything: a clean tree just reports `HEAD` as the restore point, a dirty tree is captured (committed + uncommitted + untracked) into a tag **`repair-backup-<timestamp>`** via a throwaway index (HEAD, the branch and the working tree are left untouched). The final `REPAIR_OK` line reports the ref as `backup=<tag|HEAD>`. **To recover after a repair:** review what changed with `git -C <project> diff <ref>`, and restore a clobbered file with `git -C <project> checkout <ref> -- <path>`. If the project isn't a git repo (or the snapshot fails), repair **aborts** rather than change files unprotected — `git init && git add -A && git commit` first, or pass **`--no-backup`** to skip the safety net.

### 1b. `--repair` does NOT touch `CLAUDE.md` — read this before assuming it does

**`scaffold.sh --repair` preserves `CLAUDE.md` verbatim. It is the only artifact `--repair` will not regenerate.** Reason: `CLAUDE.md` carries the user's project intentions — auto-rewriting from the template would clobber custom edits. The generators run by `--repair` (`serve-options`, `devcontainer`, `claude-settings`, `playwright`, optionally `firebase-emulators`) touch `project.json`, `.devcontainer/*`, `.claude/settings.json`, `package.json`, and the Firebase config files — never `CLAUDE.md`.

This produces an **asymmetry** that bites on toolkit upgrades:

| What | How it gets to a project | Retrofitted by `--repair`? |
|---|---|---|
| Generator outputs (configs, mounts, scripts, deps) | Generators (idempotent rewrites) | **Yes** — `--repair` re-runs them. |
| In-context `CLAUDE.md` sections (Architect mentality, Architecture-first, Firebase recipe, Playwright brief, etc.) | Step 2 of THIS skill, from `CLAUDE.md.tmpl` | **No** — hand-merge per step 2 below. |

**When you should hand-merge sections into an existing `CLAUDE.md`:**
1. After running `--repair` on a project scaffolded with an older toolkit version, and the verify checklist (step 3) flags missing CLAUDE.md sections.
2. After running `--repair --firebase` to add Firebase, since the Firebase recipe section in CLAUDE.md (the connect-to-real-project walkthrough) is only added by step 2 of this skill, not by `--repair`.

## Notes

- **Canonical source** of this skill, `scaffold.sh`, and all templates is the `claude-toolkit` repo. Edit there - never fork copies into individual projects.
- **Repair an existing project** with `scaffold.sh --repair <project-path-or-name> [<app-name>]` to re-apply the three house generators (idempotent).
- Layout is **integrated**: apps in `apps/`, libraries in `libs/`. Generate more with **`nx g @bespunky/nx-tools:app apps/<name>`** (the house app generator — it applies the per-app config, including Firebase wiring when the workspace uses it, that raw `@nx/angular:application` omits; see the *add another app* recipe below) and `nx g @nx/angular:library libs/<name>`.
- `yarn create nx-workspace` runs `git init` + an initial commit inside the container (git identity is passed through from the host); the script then commits the generator outputs on top and (unless `--no-github`) creates a **private** GitHub repo host-side via `gh` and pushes `main` to it.

### Recipe: add another app to the workspace

The scaffolder configures the **first** app, but a workspace can hold many. To add another, **always go through the house `app` generator — never raw `@nx/angular:application`** (which produces an unconfigured app: no `serve` `host: 0.0.0.0`, and in a Firebase workspace none of the per-app Firebase wiring — environment files, `firebase.config.ts`, `app.config` wiring, the production `fileReplacements`, the `platform:web` tag, or the serve orchestrator):

```
yarn nx g @bespunky/nx-tools:app apps/<new-app-name>
```

This is the **exact command 1.1 runs for the first app**, so the new app is configured identically — no drift. The generator delegates to `@nx/angular:application` with the house defaults, applies `serve-options`, and **auto-detects Firebase** from `firebase.json` at the workspace root (resolving the emulators' `demo-<workspaceName>` identity from the workspace root, not the new app's name), so a second app in a Firebase workspace is wired end-to-end with **no flag**. Override with `--firebase=false` to opt one app out, or `--firebase=true` to force wiring where `firebase.json` isn't present yet.

Run it from inside the devcontainer (where the workspace's `node_modules` holds the generators). `@bespunky/nx-tools` is project-runnable once it's published to npm (the scaffold persists it as a devDependency — same as the `mark-extractable` / `adopt-extracted` generators); until then, invoke it via the scaffolder's staging (`scaffold.sh --repair` re-stages the house tools) or from the toolkit checkout.

### Recipe: add Firebase to an existing project

If a project was scaffolded **without** `--firebase` and you now need it, use repair mode + the flag — **never hand-create the artifacts**, always go through the generator (this is the BeSpunky way: *generator-first, manual last*):

```
bash "${CLAUDE_SKILL_DIR}/assets/scaffold.sh" --repair --firebase <PROJECT_PATH_OR_NAME> [<APP_NAME>]
```

(Wrap with `wsl.exe bash -lc '…'` on Windows hosts, as in step 1.) This re-runs the house generators idempotently on the existing project; the `firebase-emulators` generator writes `firebase.json` (does **not** write `.firebaserc` — see below), generates the env files `apps/<APP>/src/environments/{environment.interface.ts, environment.ts, environment.no-emulators.ts, environment.prod.ts}` (the latter three only if absent — your customizations and real production web config are preserved), writes or self-heals `apps/<APP>/src/app/firebase.config.ts` (written when absent; rewritten when it detects the legacy `ngDevMode`-and-two-consts shape — in that case any populated `productionFirebaseConfig` values are migrated into `environment.prod.ts` before the legacy file is overwritten, with both `[firebase-emulators] Migrated ...` and `[firebase-emulators] Rewriting ... from the legacy ngDevMode shape` lines logged. Once `firebase.config.ts` is on the current shape (per-service gating + the `ngDevMode` prod-DCE guard, importing `./emulator-overrides`), the generator leaves it alone — your custom providers, `messagingSenderId`, `initializeFirestore` options, etc. are preserved across `--repair`), registers the `environment.ts` → `environment.prod.ts` `fileReplacements` on the production build configuration, best-effort-wires `app.config.ts`, scaffolds `apps/functions` + `firebase/project.json` + the emulator tooling under `tools/` (seeds, cache, reset, verified reaping), (re)shapes the serve targets into the three single-purpose targets — `serve` (orchestrator: `firebase:emulators` + `serve-with-emulators`), `serve-with-emulators` and `serve-no-emulators` (each a dev-server pinned to one build env, no sub-configs) — self-healing the older `serve-app`/`serve-standalone` names, the `serve-no-emulators` that doubled as the orchestrator's inner dev-server, and the `dependsOn: ["emulators"]`/`emulators:exec` shapes into them, inserts the platform-firewall ESLint constraints, asserts the `.gitignore` entries + `nx.json` TUI-off, and updates `package.json` (the post-commit `installPackagesTask` runs install). The `devcontainer` generator simultaneously adds the Firebase CLI + Google Cloud CLI features, the `toba.vsfire` extension, labeled `portsAttributes`, and same-port `forwardPorts` for the dev server + emulator suite. **Rebuild the devcontainer** afterwards (VS Code → *Dev Containers: Rebuild Container*) to materialize the new features.

After the retrofit, run step 3 (Verify) against the project — every check that mentions "if `--firebase` was used" must pass. If `app.config.ts` couldn't be auto-wired, the generator logged a warning; add `provideAppFirebase()` to its providers array by hand.

**Also hand-merge the Firebase section into the project's `CLAUDE.md`.** `--repair` doesn't touch `CLAUDE.md` (see §1b above), so the connect-to-a-real-project recipe + the App Hosting walkthrough that the template carries inside `{{#firebase}}...{{/firebase}}` won't appear automatically. Open `${CLAUDE_SKILL_DIR}/assets/CLAUDE.md.tmpl`, copy the rendered Firebase block, and paste it into the project's `CLAUDE.md` at the same position (between the Stack section and the Generator-first section). ### Recipe: connect a real Firebase project (App Hosting)

The scaffolder deliberately does NOT fabricate `.firebaserc` or the production web config — the cloud state is the Firebase CLI's source of truth, and inventing values would lie about what the user's account actually has and break `firebase apphosting:*` / `firebase use` the moment they're touched. The BeSpunky default is Firebase **App Hosting** (the framework-aware product), not classic static Hosting — App Hosting build/runtime config lives in `apphosting.yaml` at the workspace root (generated by the scaffolder, ready to override as needed). When you (or the user) are ready to wire a real Firebase project, run these from the workspace root inside the devcontainer (where the Firebase CLI is installed):

```
firebase login                                                            # one-time auth against the user's Google account
firebase use --add                                                        # pick a project from their account — writes .firebaserc
firebase apphosting:backends:create --project <projectId>                 # one-time: creates the App Hosting backend (interactive — LINK THIS REPO; that linkage is the deploy CI)
firebase apps:sdkconfig WEB <appId> --project <projectId>                 # prints the real web config for client-side SDK init
```

Paste the returned `firebaseConfig` fields into the `firebase` block of **`apps/<APP>/src/environments/environment.prod.ts`** — the production environment file, which the Angular build swaps in for `environment.ts` via `project.json`'s `targets.build.configurations.production.fileReplacements`. **Never edit `apps/<APP>/src/app/firebase.config.ts`** — that's structural code the generator owns end-to-end and rewrites on every `--repair --firebase`. 

**The deploy CI is Firebase's — we generate no workflow.** When the user **links the project's GitHub repo** at `firebase apphosting:backends:create` (the repo already exists — the scaffolder created it), App Hosting provisions and maintains its own Cloud Build pipeline: **pushing to the configured branch triggers a build via Cloud Build and a Cloud Run deploy.** That is the "automatic CI with Firebase deploy" — owned and kept current by Firebase's GitHub integration, so the project never carries a `.github/workflows/*` deploy file we'd have to update as Firebase evolves. Tweak build/runtime behavior in `apphosting.yaml` at the workspace root. The local emulator setup is unaffected by any of this — emulators continue to use the `demo-<workspaceName>` offline project id passed by the Nx targets.
